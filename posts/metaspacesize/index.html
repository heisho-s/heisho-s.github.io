MaxMetaspaceSize を CompressedClassSpaceSize のデフォルト値（1GB）より小さく指定した際の挙動について

-XX:CompressedClassSpaceSize はデフォルトで 1GB の値だが、-XX:MaxMetaspaceSize でこの値より小さく指定すると、 CCS のサイズがどういう挙動で確保されるのか気になったので試してみた。

背景

CompressedClassSpace は Metaspace に含まれる領域であり、デフォルトで 1GB を確保する。圧縮 OOP を利用する場合には適切な Metaspace 及び CompressedClassSpace のサイズをオプションで指定する必要があるが、-XX:CompressedClassSpaceSize の付与を忘れることにより CCS のサイズが 1GB になってしまい、Metaspace のサイズが十分に確保されず OOME が発生するケースも多々見られる。

そこで -XX:MaxMetaspaceSize のみ値を指定しているケースについて、 その指定値が -XX:CompressedClassSpaceSize のデフォルト値 1GB より小さい場合に CCS のサイズがどうなるか気になったので確認した。

確認結果

OpenJDK の各 LTS バージョンで確認したところ、以下の通りとなった。

Java バージョン-XX:MaxMetaspaceSize-XX:CompressedClassSpaceSizeCCS のおおよその計算式
Java 8104857600(100MB)96468992(92MB)MaxMetaspaceSize - 8MB
Java 11104857600(100MB)96468992(92MB)MaxMetaspaceSize - 8MB
Java 17104857600(100MB)83886080(80MB)MaxMetaspaceSize * 0.8

動作確認

-XX:MaxMetaspaceSize=100m を指定し、-XX:+PrintFlagsFinal で各領域サイズの値を確認する。

以上より Java 8 と Java 11 では確保するアルゴリズムに変更はなく、Java 17 で何かしらの変更が入っていると思われる。これを確認するため OpenJDK の実装を見ていく。

実装

Java 8

  • hotspot/src/share/vm/memory/metaspace.cpp
   3237   // Initial virtual space size will be calculated at global_initialize()
   3238   uintx min_metaspace_sz =
   3239       VIRTUALSPACEMULTIPLIER * InitialBootClassLoaderMetaspaceSize;
   3240   if (UseCompressedClassPointers) {
   3241     if ((min_metaspace_sz + CompressedClassSpaceSize) >  MaxMetaspaceSize) {
   3242       if (min_metaspace_sz >= MaxMetaspaceSize) {
   3243         vm_exit_during_initialization("MaxMetaspaceSize is too small.");
   3244       } else {
   3245         FLAG_SET_ERGO(uintx, CompressedClassSpaceSize,
   3246                       MaxMetaspaceSize - min_metaspace_sz);
   3247       }
   3248     }
   3249   } else if (min_metaspace_sz >= MaxMetaspaceSize) {
   3250     FLAG_SET_ERGO(uintx, InitialBootClassLoaderMetaspaceSize,
   3251                   min_metaspace_sz);
   3252   }

CCS のデフォルトは 1GB だが、上記の通り min_metaspace_sz + CompressedClassSpaceSize > MaxMetaspaceSize の場合は MaxMetaspaceSize - min_metaspace_sz が CCS のサイズとして設定される。 min_metaspace_sz は 3238 行目で定義されており、VIRTUALSPACEMULTIPLIER * InitialBootClassLoaderMetaspaceSize となっている。

InitialBootClassLoaderMetaspaceSize は本環境では以下の通りデフォルト 4MB である。

$ java -XX:MaxMetaspaceSize=100m -XX:+PrintFlagsFinal -version | grep -e InitialBootClassLoaderMetaspaceSize
    uintx InitialBootClassLoaderMetaspaceSize       = 4194304                             {product}

VIRTUALSPACEMULTIPLIER は metaspace.cpp 内で以下の通り定義されている。

   3002 #define VIRTUALSPACEMULTIPLIER 2

よって min_metaspace_sz は 4MB * 2 = 8MB となる。 したがって min_metaspace_sz + CompressedClassSpaceSize > MaxMetaspaceSize の場合において、CompressedClassSpaceSize は MaxMetaspaceSize - 8MB の値が指定される。

これは動作確認の結果と整合性がとれる。

Java 11

Java 8 と同様のはずなので省略。

Java 17

  • src/hotspot/share/memory/metaspace.cpp
    683   if (UseCompressedClassPointers) {
    684     // Let CCS size not be larger than 80% of MaxMetaspaceSize. Note that is
    685     // grossly over-dimensioned for most usage scenarios; typical ratio of
    686     // class space : non class space usage is about 1:6. With many small classes,
    687     // it can get as low as 1:2. It is not a big deal though since ccs is only
    688     // reserved and will be committed on demand only.
    689     size_t max_ccs_size = MaxMetaspaceSize * 0.8;
    690     size_t adjusted_ccs_size = MIN2(CompressedClassSpaceSize, max_ccs_size);
    691
    692     // CCS must be aligned to root chunk size, and be at least the size of one
    693     //  root chunk.
    694     adjusted_ccs_size = align_up(adjusted_ccs_size, reserve_alignment());
    695     adjusted_ccs_size = MAX2(adjusted_ccs_size, reserve_alignment());
    696
    697     // Note: re-adjusting may have us left with a CompressedClassSpaceSize
    698     //  larger than MaxMetaspaceSize for very small values of MaxMetaspaceSize.
    699     //  Lets just live with that, its not a big deal.
    700
    701     if (adjusted_ccs_size != CompressedClassSpaceSize) {
    702       FLAG_SET_ERGO(CompressedClassSpaceSize, adjusted_ccs_size);
    703       log_info(metaspace)("Setting CompressedClassSpaceSize to " SIZE_FORMAT ".",
    704                           CompressedClassSpaceSize);
    705     }
    706   }

想定通り実装が Java 8 や Java 11 より変更されている。 max_ccs_size として MaxMetaspaceSize の 0.8倍 の値を確保し、その値と CCS のデフォルト値を比較し小さいほうを adjusted_ccs_size として確保する。

アラインメントの調整の後、adjusted_ccs_size と CompressedClassSpaceSize が異なる場合 adjusted_ccs_size を CCS として設定する。

上記より、MaxMetaspaceSize * 0.8 が CCS のデフォルト 1GB より小さければ、 アラインメント調整で値が変わらなければ CCS のサイズは MaxMetaspaceSize * 0.8 となる。

これは動作確認の結果と整合性がとれる。

まとめ

CompressedClassSpace は Metaspace に含まれる領域なので、Metaspace < CompressedClassSpace となるようなサイズの指定をすると、矛盾を起こさないために内部でサイズ調整されていることが分かった。
